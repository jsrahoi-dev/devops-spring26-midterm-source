    ### 1. System Overview & Architecture
    <p>
      The Color Perception SPA is a research application designed to study how people
      with different native languages classify colors. Users are presented with randomly- generated colors from the RGB color space (16.7 million possible colors), and are asked classify them into one of 11 "parent colors", and can explore agreement patterns in an interactive 3D visualization.

    </p>

      <li><strong>Database:</strong> MySQL 8 on AWS RDS with InnoDB storage engine</li>
      <li><strong>Containerization:</strong> Docker with multi-stage builds (frontend build stage + production backend stage)</li>
      <li><strong>Cloud Infrastructure:</strong> AWS EC2 (t3.micro) for compute, RDS for database, ECR for container registry</li>
      <li><strong>CI/CD:</strong> GitHub Actions with semantic-release for automated versioning</li>

    ### 4. What I Built in GitHub

    #### Repository Structure
    <p>
      I created a dual-repository architecture to separate concerns between application
      code and infrastructure deployment:
    </p>
    <ul>
      <li><strong>Source Repo:</strong> Contains React frontend, Express backend, Dockerfile, semantic-release configuration</li>
      <li><strong>Infra Repo:</strong> Contains deployment scripts, integration tests, AWS resource configurations, GitHub Actions workflows</li>
    </ul>

    #### GitHub Actions Workflows Created
    <ul>
      <li><strong>release.yml (Source):</strong> Runs semantic-release on every push to main, creates version tags with conventional commit analysis</li>
      <li><strong>trigger-rc-deploy.yml (Source):</strong> Listens for RC tag creation, triggers infra repo via repository dispatch</li>
      <li><strong>rc-build.yml (Infra):</strong> Comprehensive build, test, and deploy workflow with three-tier testing</li>
      <li><strong>integration-tests.yml (Infra):</strong> Standalone integration test suite for manual validation</li>
    </ul>

    #### OIDC Setup for AWS
    <p>
      Instead of using static AWS credentials, I implemented OpenID Connect (OIDC) authentication:
    </p>
    <ul>
      <li>Created an IAM OIDC identity provider for GitHub</li>
      <li>Created GitHubActionsDeployRole with trust policy allowing GitHub Actions to assume the role</li>
      <li>Configured workflows to use aws-actions/configure-aws-credentials with OIDC role assumption</li>
      <li>Benefits: No long-lived credentials, automatic rotation, fine-grained permissions</li>
    </ul>

    #### Secrets Management
    <ul>
      <li><strong>INFRA_REPO_TOKEN:</strong> Personal access token for repository dispatch and cross-repo communication</li>
      <li><strong>AWS_ACCOUNT_ID:</strong> Used in ECR image URIs</li>
      <li><strong>DB_HOST, DB_USER, DB_PASSWORD, DB_NAME:</strong> RDS connection details</li>
      <li><strong>SESSION_SECRET:</strong> Express session signing key</li>
    </ul>

    #### Branch Protection Rules
    <ul>
      <li>Main branch protected with required status checks</li>
      <li>All changes must go through pull requests</li>
      <li>Conventional commits enforced for semantic versioning</li>
    </ul>

    ### 6. Key Implementation Details

    #### Database Migrations
    <p>
      I implemented a safe migration strategy using SQL schema files:
    </p>
    <ul>
      <li>CREATE TABLE IF NOT EXISTS prevents errors on redeployment</li>
      <li>Foreign key constraints ensure referential integrity</li>
      <li>Indexes on rgb_r, rgb_g, rgb_b for fast color lookups</li>
      <li>InnoDB engine for ACID compliance and row-level locking</li>
      <li>Migration runs automatically on container startup</li>
    </ul>

    #### Session Management
    <p>
      Session persistence was critical for tracking user responses across multiple colors:
    </p>
    <ul>
      <li>Used express-mysql-session to store sessions in RDS instead of memory</li>
      <li>Set app.set('trust proxy', 1) to handle Nginx reverse proxy headers</li>
      <li>Configured secure cookies for production (httpOnly, secure flags)</li>
      <li>7-day session expiration to allow users to return</li>
      <li>Session ID used as foreign key in responses table</li>
    </ul>

    #### Random RGB Color Generation
    <p>
      Unlike traditional systems with predefined color palettes, this app generates
      colors from the entire RGB color space:
    </p>
    <ul>
      <li>Math.random() * 256 for each channel (R, G, B)</li>
      <li>16,777,216 possible unique colors</li>
      <li>Algorithm attempts to avoid showing duplicate colors to the same user</li>
      <li>Falls back to whatever color was generated after 3 attempts (prevents infinite loops)</li>
      <li>Converts RGB to hex for CSS display (#RRGGBB format)</li>
    </ul>

    #### Three-Tier Testing Strategy
    <ul>
      <li><strong>Tier 1 (Pre-Build):</strong> Fast feedback on code quality before building containers</li>
      <li><strong>Tier 2 (Smoke Tests):</strong> Validates container works in isolation</li>
      <li><strong>Tier 3 (Integration):</strong> Tests full system with real AWS resources, catches configuration issues</li>
    </ul>

    #### Temporary EC2 Integration Testing
    <p>
      To ensure deployments work in a production-like environment without affecting RC:
    </p>
    <ul>
      <li>Provision fresh t3.micro instance with same AMI as RC</li>
      <li>Deploy container and configure RDS connection</li>
      <li>Run integration tests (session creation, color storage, API workflows)</li>
      <li>Always cleanup: terminate instance even if tests fail (uses always() in workflow)</li>
      <li>Cost-effective: instance runs for ~5 minutes per build</li>
    </ul>

    #### Docker Multi-Stage Build
    <p>
      Optimized for production with a two-stage build:
    </p>
    <ul>
      <li><strong>Stage 1:</strong> node:20 full image, builds React with Vite (npm run build)</li>
      <li><strong>Stage 2:</strong> node:20-slim, copies built frontend + backend, removes dev dependencies</li>
      <li>Final image size: ~200MB vs. ~1GB with single-stage build</li>
      <li>Faster pulls from ECR, lower storage costs</li>
    </ul>

    ### 7. Challenges & Solutions

    #### Challenge: Session Persistence Behind Nginx
    <p><strong>Problem:</strong> Sessions weren't persisting between requests. Users would lose their language selection and color history.</p>
    <p><strong>Solution:</strong> Added app.set('trust proxy', 1) to trust Nginx's X-Forwarded-For headers. This fixed session cookie issues.</p>

    #### Challenge: YAML Syntax in GitHub Actions
    <p><strong>Problem:</strong> Workflows failed with cryptic YAML parsing errors, especially with multiline strings in deployment scripts.</p>
    <p><strong>Solution:</strong> Used | for literal block scalars and proper indentation. Validated YAML with online linters before committing.</p>

    #### Challenge: Semantic-Release Token Permissions
    <p><strong>Problem:</strong> Semantic-release couldn't create tags with GITHUB_TOKEN - kept getting 403 errors.</p>
    <p><strong>Solution:</strong> Created a Personal Access Token (INFRA_REPO_TOKEN) with repo and workflow scopes. Used persist-credentials: false in checkout.</p>

    #### Challenge: Database Migration Safety
    <p><strong>Problem:</strong> Needed migrations to run on every deployment but not fail if tables already existed.</p>
    <p><strong>Solution:</strong> Used CREATE TABLE IF NOT EXISTS and idempotent migration scripts. Tested migration re-runs in integration tests.</p>

    #### Challenge: Integration Test Cleanup
    <p><strong>Problem:</strong> Failed integration tests left orphaned EC2 instances running, costing money.</p>
    <p><strong>Solution:</strong> Wrapped cleanup in always() condition in GitHub Actions. Also added tags to instances for manual cleanup scripts.</p>

    #### Challenge: Docker Build Context Size
    <p><strong>Problem:</strong> Docker build was slow because it was copying node_modules to build context.</p>
    <p><strong>Solution:</strong> Added .dockerignore file excluding node_modules, .git, and other unnecessary files. Build time dropped from 5min to 2min.</p>

    ### 8. What I Would Do Next (Production)

    <p>
      The current RC environment is production-ready in terms of code quality and testing,
      but lacks the scalability, reliability, and security features needed for a real
      production deployment. Here's my roadmap:
    </p>

    #### Production Environment Setup
    <ul>
      <li>Separate production EC2 instance(s) with different security groups</li>
      <li>Private subnets for EC2 and RDS (no public IPs)</li>
      <li>NAT Gateway for outbound internet access from private subnets</li>
      <li>Bastion host or Systems Manager Session Manager for secure SSH access</li>
      <li>Separate RDS instance for production with Multi-AZ deployment</li>
    </ul>

    #### Blue-Green Deployment
    <ul>
      <li>Two production environments (blue and green)</li>
      <li>Deploy to inactive environment, test, then switch traffic</li>
      <li>Instant rollback by switching back to previous environment</li>
      <li>Zero-downtime deployments</li>
      <li>Implemented via Application Load Balancer target group switching</li>
    </ul>

    #### Auto-Scaling with Application Load Balancer
    <ul>
      <li>Application Load Balancer (ALB) to distribute traffic across multiple EC2 instances</li>
      <li>Auto Scaling Group with min 2, max 10 instances based on CPU/memory metrics</li>
      <li>Health checks at /api/health with automatic instance replacement</li>
      <li>Sticky sessions to maintain user sessions during scale operations</li>
      <li>SSL/TLS termination at ALB with ACM certificate</li>
    </ul>

    #### CloudFront CDN
    <ul>
      <li>Global content delivery network for frontend assets</li>
      <li>Cache static assets (JS, CSS, images) at edge locations</li>
      <li>Reduce latency for international users</li>
      <li>DDoS protection via AWS Shield Standard (included with CloudFront)</li>
      <li>Custom domain with HTTPS (e.g., colorperception.rahoi.dev)</li>
    </ul>

    #### Route53 DNS
    <ul>
      <li>Custom domain management</li>
      <li>Health checks with failover to backup region</li>
      <li>Weighted routing for gradual rollouts (10% of traffic to new version)</li>
      <li>Geolocation routing for region-specific deployments</li>
    </ul>

    #### CloudWatch Monitoring & Alarms
    <ul>
      <li>Application metrics: Request count, latency, error rates</li>
      <li>Infrastructure metrics: CPU, memory, disk usage, network throughput</li>
      <li>Database metrics: Connection count, query latency, replication lag</li>
      <li>Custom metrics: Color classifications per minute, session creation rate</li>
      <li>Alarms with SNS notifications (email/SMS) for:
        <ul>
          <li>Error rate &gt; 5%</li>
          <li>Latency P95 &gt; 1000ms</li>
          <li>Database CPU &gt; 80%</li>
          <li>Disk usage &gt; 85%</li>
        </ul>
      </li>
      <li>CloudWatch Logs for centralized log aggregation</li>
      <li>Log insights queries for debugging and analytics</li>
    </ul>

    #### Cost Optimization
    <ul>
      <li>Reserved Instances for baseline capacity (40-60% savings vs on-demand)</li>
      <li>Spot Instances for auto-scaling overflow (up to 90% savings)</li>
      <li>RDS reserved instances for 1-year term</li>
      <li>S3 lifecycle policies for old container images (move to Glacier after 90 days)</li>
      <li>CloudWatch Logs retention policies (30 days for debug logs, 1 year for audit logs)</li>
      <li>Compression for CloudFront assets (gzip/brotli)</li>
      <li>Database query optimization (reduce RDS instance size)</li>

    #### Disaster Recovery
    <ul>
      <li>RDS automated backups with 30-day retention</li>
      <li>Manual snapshots before major deployments</li>
      <li>Point-in-time recovery within backup window</li>
